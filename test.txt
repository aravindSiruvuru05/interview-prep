

To clarify the relationship between CPU, core, process, and thread, let's break down each term:
1. CPU (Central Processing Unit):
The CPU is the "brain" of the computer. It's a hardware component that executes instructions from computer programs.  
It performs arithmetic, logical, control, and input/output (I/O) operations.
2. Core:
A core is an individual processing unit within a CPU. Modern CPUs often have multiple cores (e.g., dual-core, quad-core, octa-core).  
Each core can execute instructions independently, allowing the CPU to perform multiple tasks simultaneously.  
Think of cores as individual "mini-CPUs" within the main CPU.
3. Process:
A process is a running instance of a computer program. When you open an application, the operating system creates a process for it.
A process has its own dedicated memory space and resources.
Processes are independent of each other, meaning that if one process crashes, it typically doesn't affect other processes.  
4. Thread:
A thread is the smallest unit of execution within a process. A process can have multiple threads.  
Threads share the same memory space and resources as their parent process.  
Threads allow a process to perform multiple tasks concurrently. For example, a web browser might have one thread for downloading a webpage and another thread for displaying it.  
Essentially a thread is a sequence of programmed instructions that can be managed independently by the operating system.  
In simpler terms:
The CPU is the overall worker.
Cores are the individual workstations within that worker.
A process is a complete task that the worker is doing.
Threads are the individual steps within that task.
Key Relationships:
A CPU contains one or more cores.  
A process runs on a CPU core.
A process can contain one or more threads.  
Multiple threads can run concurrently on multiple cores.

The way a Go program or any application utilizes CPU cores, processes, and threads is a bit nuanced and depends on several factors:
1. Basic Go Program (Single Goroutine):
If you write a simple Go program with just the main function and no explicit concurrency (no goroutines), it will run in a single operating system thread, which will be scheduled onto a single CPU core.
This means it's running as a single process with a single thread.
The operating system handles the scheduling, so which core it runs on can change over time.  
2. Go Programs with Goroutines:
Go's power comes from goroutines. Goroutines are lightweight, concurrently executing functions.  
When you create goroutines, the Go runtime uses a scheduler to distribute them across available operating system threads.  
The number of operating system threads used by the Go runtime is determined by the GOMAXPROCS environment variable. By default, GOMAXPROCS is set to the number of CPU cores available.
Therefore, goroutines can run concurrently on multiple cores, effectively utilizing the CPU's processing power.  
The Go runtime's scheduler maps goroutines to operating system threads. And the OS maps those threads to the CPU cores.  
So, a Go program with many goroutines can utilize multiple cores.  
3. Other Applications:
Single-threaded applications: Some applications are designed to run in a single thread. These will typically use only one core at a time.
Multi-threaded applications: Modern applications often use multiple threads to improve performance. For example:
Web browsers might use separate threads for rendering web pages, executing JavaScript, and downloading resources.  
Video editing software might use multiple threads for encoding and decoding video.  
Operating systems handle the scheduling of threads to cores.
Processes: Every application runs in its own process. This isolates the application's memory and resources from other applications.
Operating System Scheduling: The operating system's scheduler is responsible for distributing processes and threads across available CPU cores. It aims to provide fair and efficient use of the CPU.  
In essence:
A simple, non-concurrent program will use a single core.
Go programs with goroutines can utilize multiple cores. 
Most modern applications are multi-threaded and take advantage of multiple cores.  
The operating system manages the scheduling of processes and threads to cores.  
Therefore, whether an application uses a single core or multiple cores depends on its design and the operating system's scheduling.

What is a Thread?
A thread is the smallest unit of execution within a process. It represents an independent sequence of instructions that can run concurrently with other threads. Threads allow programs to perform multiple tasks at the same time, improving performance and responsiveness. When a process starts, memory and resources are allocated—each thread in the process shares that memory and resources, except the stack. Every thread has it's own stack. A single-threaded process can perform one task at a time, but a multi-threaded process can perform multiple tasks at the same time.

Types of Threads:
User Threads: Managed at the application level without direct kernel involvement. (e.g., Java Threads, POSIX Threads)
Kernel Threads: Managed directly by the operating system, allowing better CPU scheduling.
Multithreading: The ability of a CPU or a program to execute multiple threads concurrently.
Challenges of Multithreading:
Race conditions – When multiple threads access shared data without synchronization, leading to unpredictable results.
Deadlocks – When two or more threads wait for resources indefinitely.
Complex debugging – Harder to trace issues due to concurrency.


Context Switching in Computer Science
Context switching is the process of saving the state of a currently running thread or process and restoring the state of another thread or process so that it can resume execution. This allows a CPU to switch between multiple processes or threads efficiently, enabling multitasking and concurrency in modern operating systems.


When Does Context Switching Occur?
Multitasking: The OS switches between multiple processes to share CPU time.
Multithreading: The CPU switches between threads of the same process.
Interrupt Handling: When an interrupt occurs (e.g., I/O completion), the OS may switch to handling it.
Preemptive Scheduling: The OS may forcibly switch processes based on scheduling policies (e.g., Round Robin).

What Are Logical Cores?
Logical cores refer to the processing units available to the operating system for executing tasks. These can be physical CPU cores or hyper-threaded virtual cores.
Physical Cores: Actual cores in a CPU.
Logical Cores: If a CPU supports Hyper-Threading (Intel) or Simultaneous Multi-Threading (AMD), each physical core can handle two (or more) execution threads, creating additional logical cores.
For example:
A 4-core CPU without hyper-threading = 4 logical cores.
A 4-core CPU with hyper-threading (2 threads per core) = 8 logical cores.
How It Relates to Go
Go's runtime heavily depends on logical cores for its concurrency model.
1. GOMAXPROCS (Goroutine Scheduling)
Go schedules goroutines (lightweight threads) across logical cores.
The GOMAXPROCS environment variable (or runtime.GOMAXPROCS(n)) controls how many OS threads Go runs concurrently.
By default, GOMAXPROCS is set to the number of logical cores.
2. Goroutines and OS Threads
Goroutines are not mapped 1:1 to OS threads.
The Go scheduler manages multiple goroutines over a fixed number of OS threads (M:N scheduling model).
More logical cores increase parallelism (goroutines running truly in parallel).
3. Performance Impact
If you increase GOMAXPROCS beyond logical cores, it won't improve performance significantly (may even hurt due to context switching).
If your Go application is CPU-bound (e.g., video encoding, cryptography), more logical cores help.
For I/O-bound applications (e.g., web servers), goroutines efficiently handle tasks even with fewer cores.

How Go Handles API Calls in Goroutines
You start a goroutine to make an API call.
The API request is sent by the OS kernel (through the network stack).
The goroutine waits for the response, but the Go runtime schedules other goroutines on available threads.
When the response arrives, the goroutine resumes execution.
✅ Other goroutines keep running without blocking because Go's scheduler automatically assigns them to available OS threads.
Where Does the Waiting Happen?
The actual API call is performed by the OS, not Go.
The goroutine is waiting (blocked) inside the Go runtime, but Go runtime does not block OS threads.
The Go scheduler switches execution to other goroutines while waiting.
How Go Avoids Blocking OS Threads
Go uses M:N scheduling:
Many goroutines (M) are mapped to a few OS threads (N).
When a goroutine performs an I/O operation (like an API call), the Go runtime parks the goroutine and frees the OS thread.
Another goroutine is scheduled to run on the freed OS thread.
When the API response arrives, the parked goroutine resumes execution.

If millions of goroutines are suddenly triggered, your system might not immediately crash, but its performance will depend on several factors. Let’s break this down:

How Go Handles Millions of Goroutines?
🔹 Goroutines are lightweight (2 KB initial stack size). Unlike OS threads, they don’t consume large memory or scheduling overhead.
🔹 Go runtime uses an M:N scheduler, meaning multiple goroutines share fewer OS threads efficiently.
🔹 If a goroutine is waiting on I/O (like API calls), it doesn’t block an OS thread, allowing more goroutines to run.
So, creating millions of goroutines won’t instantly crash the system, but there are potential issues:

Potential Bottlenecks & Issues
1️⃣ Memory Exhaustion (Primary Risk)
Each goroutine takes at least 2 KB of stack space.
1 million goroutines = ~2 GB RAM minimum. 10^6 * 2kb -> 1 * 2gb
If you spawn too many, your system may run out of memory and crash.
2️⃣ CPU Overload
If all goroutines do heavy computation, they will compete for CPU time.
Even though Go distributes work across CPU cores, CPU-bound tasks can slow everything down.
3️⃣ Network Bottleneck
If all goroutines are making API calls, your network bandwidth and remote API rate limits might become a bottleneck.
Too many API calls might lead to timeouts, failures, or rate limiting.
4️⃣ Go Scheduler Overhead
Even though Go handles scheduling well, managing millions of active goroutines still adds scheduling overhead.
If there are too many runnable goroutines, context switching can slow down execution.

What Happens in Your Case?
Let's say millions of goroutines are making API calls:
Goroutines are created (this consumes RAM).
API calls are sent to OS networking stack.
While waiting, goroutines get parked, freeing OS threads.
When responses come back, goroutines wake up and process data.
If too many responses arrive at once, CPU might struggle to process them all.
If RAM runs out, your system crashes due to OOM (Out of Memory) error.
P, M, and G in Go Runtime
1️⃣ G - Goroutine
Represents a lightweight user-space thread.
Each Goroutine (G) has its own stack (~2 KB initial size) and registers.
The Go runtime schedules goroutines on available P (processors).
If a goroutine is waiting (e.g., on I/O), it is parked (not consuming CPU).
2️⃣ M - Machine (OS Thread)
Represents an OS thread.
Each M is mapped to an actual OS thread (like pthread in Linux).
M executes Goroutines (G) assigned to it by P (Processor).
Multiple Ms can run in parallel on multi-core systems.
If an M gets blocked (e.g., system call), Go may create a new M to keep execution going.
3️⃣ P - Processor (Logical Processor)
Acts as a scheduler that assigns G (Goroutines) to M (OS Threads).
There are GOMAXPROCS number of Ps (by default, set to the number of CPU cores).
P maintains a run queue of goroutines (G), picking them for execution on M.

How P, M, G Work Together?
Go creates P (logical processors), usually equal to CPU cores.
Each P has a local queue of goroutines (G).
An OS thread (M) picks a P and runs its G one by one.
If an M is blocked (e.g., for I/O), another M may take over.
If P's local queue is empty, it can "steal" G from another P.
The runtime dynamically adjusts the number of M to prevent blocking.

1️⃣ Node.js Event Loop (Single-Threaded with Libuv)
Node.js uses a single-threaded event loop that runs on a single OS thread.
When an I/O operation (like an API call) occurs, the event loop offloads it to the OS (via libuv).
The OS handles the operation asynchronously.
When the operation completes, a callback is pushed to the event queue, waiting for execution.
The event loop picks up tasks from the event queue one by one, executing them on the main thread.
🔹 Key Point: Node.js is highly efficient for I/O-heavy tasks because it doesn’t block the main thread, but it struggles with CPU-heavy tasks since everything runs on a single core (unless using worker threads).

2️⃣ Go's Goroutine Model (M:N Scheduler)
Go uses goroutines, which are lighter than OS threads.
The Go runtime has its own M:N scheduler that maps multiple goroutines (G) to fewer OS threads (M).
If a goroutine performs an I/O operation (e.g., an API call), it does NOT block the thread.
Instead, Go parks that goroutine and schedules another one on the same thread.
When the I/O operation completes, the goroutine is resumed on an available thread.
Unlike Node.js, Go can utilize multiple CPU cores efficiently by running multiple OS threads in parallel.
🔹 Key Point: Go's goroutines allow true parallel execution on multi-core CPUs, while Node.js relies on a single thread (unless explicitly using worker threads).

What is a Process?
A process is an instance of a running program. It includes:
Code (Executable instructions)
Memory (Heap, Stack, Data Segments)
File descriptors, environment variables, etc.
One or more OS-managed threads
Each process is isolated and managed by the OS kernel. This isolation means:
Processes cannot directly share memory (they communicate via IPC mechanisms like pipes, shared memory, etc.).
Each process has its own virtual memory, which must be saved/restored during context switches.

1️⃣ Why is OS Process Switching Heavy?
When the OS switches from one process to another, it must:
Save the state of the current process
Registers (Program Counter, Stack Pointer, etc.)
Memory (Page Table mappings)
CPU cache (which might get flushed)
Open file descriptors
Load the state of the next process
Restore memory mappings
Reload registers
Restore execution context
Perform a context switch at the OS level
This involves a transition from user mode → kernel mode → user mode
Expensive due to cache invalidation and TLB (Translation Lookaside Buffer) flushes
💡 Why is this costly?
Each process has its own memory space → The OS must switch memory mappings.
Crossing the user-kernel boundary takes extra CPU cycles.
Cache misses occur when switching between processes.

2️⃣ Why is OS Thread Switching Less Costly than Process Switching?
A thread is a lightweight unit of execution within a process.
Threads in the same process share memory (heap, data, code).
Each thread has its own stack, registers, and instruction pointer.
When switching between threads of the same process:
The OS does not need to switch memory mappings (no TLB flush).
Only the registers, program counter, and stack need to be saved/restored.
Less overhead compared to process switching.
However, switching between threads of different processes is almost as costly as process switching.
💡 Why is this still expensive?
The OS scheduler manages thread switching → it involves kernel mode operations.
If many threads are created, OS overhead increases.

3️⃣ Why is Goroutine Switching Cheaper than OS Threads?
Goroutines are user-space threads, managed by the Go runtime scheduler (not the OS).
The OS only sees a small number of actual OS threads running goroutines.
The Go runtime handles scheduling without kernel intervention.
How Go Schedules Goroutines Efficiently?

Maps M goroutines onto N OS threads dynamically.
If a goroutine blocks (e.g., I/O operation), Go parks it and schedules another without OS involvement.

OS doesn’t need to know the state of each goroutine.
Go runtime saves/restores only the goroutine's registers and stack, avoiding OS overhead.

4️⃣ How Does Go Execute Goroutines on OS Threads?
Even though goroutines run on OS threads, the OS does not track them individually. Instead:
The Go runtime manages the goroutine state (registers, stack, etc.).
The OS only sees the Go runtime using a few OS threads.
When a goroutine blocks, Go automatically switches another goroutine onto the same OS thread.

 🚫 No!
The OS only schedules the OS thread.
The Go runtime saves/restores goroutine state in user space, avoiding OS overhead.

Does Node.js Always Use the Same Thread?
✅ Yes, for JavaScript execution
When a Node.js program starts, it runs entirely on a single thread (the main thread, part of the Event Loop).
This single thread runs synchronous JavaScript code and schedules asynchronous operations.
✅ No, for async operations (like I/O, file system, APIs, etc.)
When you make an API call, file read/write, or a DB query, Node.js delegates these tasks to worker threads via:
Libuv Thread Pool (for file system, DNS, crypto, etc.)
OS Kernel Threads (for network I/O, timers, and async tasks)
📝 Key takeaway: The main JavaScript execution always happens on a single thread, but async tasks use worker threads (managed by Node.js, not your code).
How Do Worker Threads Work in Node.js?
The main JavaScript execution stays on one thread.
Worker threads (introduced in Node.js v10) allow running CPU-heavy tasks in separate threads.
You must explicitly create worker threads using the worker_threads module.
🚀 Example: Using Worker Threads in Node.js
const { Worker, isMainThread, parentPort } = require('worker_threads');

if (isMainThread) {
  // Main thread creating a worker
  const worker = new Worker(__filename);
  worker.on("message", (msg) => console.log("Message from worker:", msg));
} else {
  // Code running in a worker thread
  parentPort.postMessage("Hello from worker!");
}

Without worker threads, Node.js remains single-threaded for JavaScript execution
Does a Goroutine Stay Attached to an OS Thread While Waiting?
🔹 No, a goroutine does NOT stay attached to an OS thread while waiting for a network call to complete.
 🔹 Instead, it gets detached from the OS thread and is put into a waiting state, freeing up the OS thread for other goroutines.

What Exactly Happens When a Goroutine Makes a Network Call?
🏁 Step 1: Goroutine Starts a Network Call
resp, err := http.Get("https://example.com")

A goroutine is created and assigned to an OS thread.
The Go net/http package makes a non-blocking system call to the OS (e.g., send() in Linux).
🚀 Step 2: Go Detaches the Goroutine & Frees the Thread
The network call is delegated to the OS, which will handle it asynchronously.
The goroutine is detached from the OS thread and put into a waiting state in Go’s scheduler.
The OS thread is now free to execute other goroutines.
🕰 Step 3: Go's Netpoller Monitors the OS
Go's netpoller (which uses epoll on Linux, kqueue on macOS/BSD, and IOCP on Windows) monitors the OS for network events.
The goroutine remains in the waiting queue without blocking any OS thread.
🔄 Step 4: Response Arrives & Goroutine Gets Rescheduled
The OS signals Go’s netpoller when the network response is ready.
The Go scheduler wakes up the waiting goroutine.
The goroutine is attached to an available OS thread (not necessarily the same one it started on).
Execution continues from where it left off.

Why Does Go Detach the Goroutine?
🔹 If Go kept the goroutine attached to an OS thread while waiting, that thread would be wasted.
 🔹 By detaching it, Go allows the OS thread to do other work while waiting.
This is what makes Go’s concurrency model highly efficient—it allows thousands (or even millions) of goroutines to run efficiently on a small number of OS threads.

Summary
✅ A goroutine detaches from the OS thread while waiting for a network call.
 ✅ The OS handles the network request asynchronously.
 ✅ Go's netpoller monitors for completion without blocking OS threads.
 ✅ Once the response arrives, the goroutine is reattached to an available thread and continues execution.
🔹 This approach prevents blocking OS threads, making Go highly scalable! 🚀

Clients have unique source IPs & ports, and
The service has enough file descriptors,
it can handle far more than 65,536 connections.

🔹 Step 6: What Limits Will You Hit First?
Before reaching port exhaustion, your service will likely hit:
File Descriptor Limits (ulimit -n).
Memory Limits (each connection consumes RAM).
Network Bottlenecks (CPU & NIC limits).
Load Balancer Constraints (some cloud LBs have max connection limits).

🔹 Step 7: How to Scale Beyond These Limits?
Increase File Descriptor Limits (ulimit -n)
Example: Increase to 5M per pod.
Distribute Load Across More Pods
Instead of 3 pods, run 10+ pods.
Enable Keep-Alive & Connection Pooling
Reuse connections to avoid new TCP handshakes.
Optimize Kernel TCP Settings
Enable net.core.somaxconn to increase backlog size.
Tune tcp_max_syn_backlog for handling bursts.
Use Multiple Listening Ports (Sharding Connections)
Instead of a single port (443), listen on multiple ports (e.g., 443, 444, 445, ...).

🔹 Final Summary
Each pod can handle connections up to its FD limit (~1M).
With 3 pods, total concurrent connections = 3M.
If requests are short-lived, RPS can be much higher (~30M RPS).
Before running out of ports, you will hit FD or memory limits first.
Scaling involves increasing FDs, adding more pods, and optimizing TCP settings.
